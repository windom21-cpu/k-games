# 引き継ぎ書 — スターマイマイ

最終更新: 2026-07-22 / 現行バージョン: v1.5.0

---

## 概要

カタツムリ(マイマイ)を育てて1年間のレースを戦う育成シミュレーション。スターホース風の「育成 → 出走 → 引退 → 配合(繁殖)」サイクルを、マイマイ題材でコミカルに翻案したもの。

- サイコロ10個をふって生まれた1頭を、半月きざみの1年間(24ターン)で調教
- 年間12レース(G3→G2→G1)を任意に選んで出走し、賞金をかせぐ
- 最終目標は12月後半の最高峰レース **「マイマイカップ」** 制覇
- 引退した個体は **殿堂** に永久保存され、**配合(v1.1 で実装済み)** の親としていつでも使える
- 牧場(スロット)単位で、レースごとのレコードタイムと通算勝利回数を蓄積(v1.2)
- 配合の遺伝は v1.3 で刷新。ステごとに優勢な親を引く方式で「父ゆずり／母ゆずり」の個性が出る

公開URL: https://windom21-cpu.github.io/k-games/star-maimai/
QRコード: `star-maimai/qr.png`

## 技術スタック

- 純粋な HTML5 + CSS + JavaScript (バニラ JS、依存ゼロ)
- 描画:
  - 画面切替は DOM ベース(`.screen.active` クラス)+ モーダル(`.modal-back.active`)
  - **モーダルの閉じ方は3系統(v1.4)**: 枠外タップ / 見出し右の `.modal-x`(✕)/ Esc キー。`wireModalDismiss()` が起動時に全 `.modal-back` へ一括配線するので、モーダルを追加しても `.modal-head` + `.modal-x` の形にしておけば自動で効く。枠外タップの判定は `e.target === back`(中身のタップでは閉じない)
  - 長い中身は `.modal-pane`(`max-height:58vh` + スクロール)に入れる。見出し・タブ・とじるボタンはスクロール域の外に残るので、**閉じる手段を探して下までスクロールする必要がない**
  - タイトルアニメ / 厩舎のマイマイ / レース実況 / 引退式は **Canvas 2D**(`drawSnail()` が共通のカタツムリ描画関数。カラの値で殻が大きく・キラキラになる)
- 入力: click / タッチ(`touch-action: manipulation`)
- 効果音: Web Audio API オシレータ(`SFX` オブジェクト)。🔊/🔇 トグルは `localStorage["starMaimai.mute"]` に永続化
- 永続化: `localStorage["starMaimai.slot0..2"]`(下記セーブ仕様)
- ビルド工程なし。`index.html` を編集して push するだけで反映 (1〜2分)

### マイマイの能力(4ステ+隠し)

| 表示 | 意味 | 上限 | 育て方 |
|---|---|---|---|
| ぬめり | 速さ(基本走力) | 120 | 調教「ガラス板すべり」 |
| 体力 | スタミナ。長距離でバテにくい | 120 | 調教「ながい散歩」 |
| 根性 | 残り24%のスパート+バテ後のねばり | 120 | 調教「タカとにらめっこ」 |
| カラ | カラカラ(乾いた)地面への耐性 | 120 | **調教では伸びない。カルシウムのエサのみ** |

ランク表示: F(<22) E D C B A S(≥110)。

**隠しステータス(画面に出さない)**:
- `hidden.gf` — 成長係数。サイコロ合計(10〜60)から 0.90〜1.25
- `hidden.apt.{num,sta,gut,shl}` — **調教効果**。ステ別に gauss(1.0, 0.25) を 0.6〜1.7 にクランプ。調教の伸び = 基本値 × apt × gf × 疲労補正。**サイコロが悪くても apt が高ければ大化けする**(ユーザー要望)
- ヒントは (1) 誕生時のコーチコメント(apt 平均×gf の水準で5段階の台詞)、(2) 調教時の「ものすごい伸びだ!」等の実感コメントのみ

### 誕生(サイコロ10個)

- 青3個→ぬめり(合計×3.0)、緑3個→体力(×3.0)、橙2個→根性(×3.5)、紫2個→カラ(×3.5)
- 合計値が成長係数 gf にも影響(=良い目は初期値と成長の二重取り)
- 名まえは手入力(最大8文字)or「おまかせ名まえ」生成

### ターン(半月×24)

毎ターン、**エサを選んで行動1つ**:
- エサ: ふつう(無料) / カルシウム(200G, カラ+2) / 特選カルシウム(500G, カラ+4・体力+1)。資金不足時は自動でふつうに降格
- 行動: 調教3種 / 休養(つかれ-50, 調子上向き) / レース出走(開催ターンのみ)
- つかれ: 調教+17前後、レース+22、80以上で調教失敗35%。55以上でレースの速度ペナルティ
- 調子: 5段階(ぜつ不調〜ぜっ好調)。毎ターン確率でゆらぐ。レース速度 ×0.90〜×1.09

### 天候・地面(マイマイの本領)

月別の雨確率(6月高・7〜8月低)+ レース毎の `wetBias` で抽選:
- **ぬれぬれ**(☔): 全員 ×1.09。カラ無関係 → 実力勝負・番狂わせの舞台
- **しっとり**(☁): ほぼ等倍
- **カラカラ**(☀): 速度 ×0.76〜1.00(カラ0→120で線形)+ スタミナ消費 ×1.3(カラで軽減)。**カラを鍛えた個体だけが晴れでも強い**(ユーザー要望どおり)

### レースシミュレーション

`stepRace()` を dt きざみで回す物理風モデル(スキップ時は `fastForwardRace()` で 0.1s きざみ一括計算 — 実況と同一ロジック):
- 速度 = (2.1 + ぬめり×0.03) × 地面係数 × 調子係数 × 疲労係数 × ゆらぎノイズ
- スタミナ = 55 + 体力×2.1。毎秒 6.5 消費(カラカラで増、スパート中×1.35)。切れると速度×(0.52+根性×0.0013)=バテ
- 残り24%でスパート: ×(1 + 根性×0.002×乱数)
- スタートに 0〜0.7s の出遅れ乱数
- 実況: ×1/×4倍速トグル、結果へスキップ、ぬめり跡・バテ表示・🥇🥈🥉

### 引退・殿堂

- 24ターン経過で自動引退。途中引退も可(確認モーダル、取消不可)
- 殿堂エントリに `canBreed: true` を必ず付与 — **引退個体は一生配合可能**(ユーザー方針)。`hidden`(gf/apt)もスナップショット保存され、遺伝計算に使われる
- 資金・殿堂・記録はスロット内で世代をまたいで引き継ぎ。引退→迎え方選択へ(年数+1)
- 引退式には「タイトルへもどる」ボタンあり(v1.1.1)

### 配合・世代交代(v1.1 / 遺伝は v1.3 で刷新)

- 殿堂に1頭でもいれば、新しいマイマイを **配合** でつくれる(野生スタートも選べる)
- 親選び: 殿堂の `canBreed` 個体+「🌿野生のマイマイ」カード(画面に入るたびに能力を引き直し。ステは `wildStatMin..Max`=25〜70、apt は新規 gauss)から **2頭をタップ選択**。3頭目をタップすると古い選択が外れる
- 遺伝(`makeSnailFromDice(dice, name, breed)` / パラメータは `TUNE.breed`):
  - **優勢な親の抽選(v1.3 の肝)**: ステごとに 50/50 でどちらの親の血が濃く出るかを引き、`statDomWeight`(0.80)だけその親へ寄せた値 `pMix` をつくる。**両親を単純平均すると「ぬめり型 × カラ型」のような組み合わせで特化が打ち消され、遺伝している感が消えてしまう**ための対策。同じ両親からでも「父ゆずりのぬめり型」「母ゆずりのカラ型」と兄弟の個性が分かれる。優勢な親が 50/50 なので**期待値は両親平均のまま = インフレしない**
  - **初期ステ加算** = 総量 `sum(pMix) × statInheritRate`(0.15)`× 変異 gauss(1, 0.35)`[0.2〜1.9] を、`pMix ^ statProfilePow`(3)`× ゆらぎ` の比で4ステに配分。**変異は総量に1回だけかける**(ステごとに振ると配分の signal がゆらぎに埋もれて個性が消える)。結果は「両親ゆずりのちから」としてサイコロ画面に表示
  - **apt(隠し調教効果)** = 同じ優勢親に `statDomWeight` で寄せた値を 25% だけ 1.0 へ回帰させ、中心に `gauss(±0.18)` → 見えるステと血筋の向きがそろう。強い血統は維持されつつインフレしない。**変異幅広め=弱親から大物が出る余地**(ユーザー指定)
  - **gf(成長係数)** = サイコロ由来×0.55 + 両親平均×0.45 + `gauss(0, 0.03)`(0.85〜1.32)
  - 世代 `gen` = max(親の gen)+1(野生親は gen 0)、`parents` に両親名を保存
- 殿堂・厩舎に世代バッジ(第n世代)と両親名を表示
- **遺伝の見せ方(v1.3)** — 遺伝の主役は apt/gf なのに完全な隠しステで、配合児でも野生児と同じ台詞しか出ず「遺伝している感」がなかったための対策:
  - サイコロ画面の「両親ゆずりのちから」で**一番濃く受けついだステをハイライト**し、直下に `✨ ○○は □□ ゆずり!`(□□=そのステで優勢だった親の名)を表示
  - **配合の子だけに出るコーチの2行目**(`breedComment()`)。子の apt 平均+gf を両親と比べ、4段階の台詞(両親をこえた / しっかり受けついだ / 少しうすまった / 両親とはちがう子)で伝える。**隠しステの遺伝をプレイヤーが知覚できる唯一の窓口**
  - `lastBreedBonus`(4ステの加算値)と `lastBreedNote`(濃く出たステ・優勢親名・apt/gf の親子比較)は表示用のモジュール変数。**セーブしない**
  - 親名は殿堂に残るユーザー入力なので、表示は必ず `esc()` を通す(`innerHTML` に差し込むため)

### 王者血統(v1.5)

**両親ともマイマイカップ制覇者**(`p1.mcWon && p2.mcWon`)のときだけ発動する特典。子に `royal: true` が立ち、引退時も殿堂へ引き継がれる。野生親は `mcWon` を持たないので王者血統にはならない。

- **カラ初期 +`mcShlBonus`(15)** — 「両親ゆずりのちから」とは別枠で加算し、誕生画面にも別行で出す
- **`mcAptRegress`(0.05)** — 隠しステ apt の平均回帰を通常(0.25)からゆるめ、**王者の素質がほぼ薄まらずに伝わる**
- 称号表示: 誕生画面(`👑 王者どうしの配合` / `👑 王者の血 — カラ +N`)、厩舎の名前よこ、殿堂・配合の親カードに `.royal-badge`、誕生ログ、`breedComment()` の王者専用台詞

**なぜカラと素質なのか(v1.5 の設計判断)**: 5案を 250試行×5世代で比較したところ、**勝率に直結するのは「見えるステの初期値」だけ**と分かった。全ステ +10 は第5世代で MC 制覇率 +14pt と明確にインフレした一方、カラ・初期資金・apt 継承はいずれも誤差の範囲だった。カラは晴れ地面でしか速度に効かずマイマイカップは雨寄り(`wetBias` 0.2 → 晴れは約31%)、しかも上限120で頭打ちになるため自己制限的。**報酬を「勝つ力」ではなく「120×4 を目指す楽しさ」の軸に置いたのが要点**。

| 案(当時の条件=片親でもMC勝者) | 第1 | 第2 | 第3 | 第4 | 第5 |
|---|---|---|---|---|---|
| ボーナスなし | 20.0 | 41.6 | 55.6 | 60.8 | 62.4 |
| カラ初期 +15 | 20.0 | 48.0 | 54.4 | 60.8 | 60.8 |
| 初期資金 +2500G | 14.4 | 49.6 | 56.0 | 58.0 | 63.6 |
| apt 平均回帰なし | 18.4 | 51.2 | 60.0 | 57.2 | 63.2 |
| **全ステ初期 +10** | 21.6 | 52.4 | 62.4 | 72.4 | **76.4** |

> 第1世代は親がいないので必ずボーナス非適用。**ここの数字のばらつきがそのまま誤差の目安**になる(250試行で約7pt、700試行で約2.5pt)。差を読むときは必ず第1世代と見くらべること。

**却下した案: 調教の疲労軽減(17→14)** — 第5世代で 44.4% と逆に下がった。疲れにくくなった結果 `fatigue < 78` の出走判定を通りやすくなり、出走数が増えて調教ターンが減るという遠回りの効き方をしていた。**効果の向きがプレイ方針しだいで変わるのでボーナスとして読みにくい**。

**実装後の実測(700試行×5世代、条件=両親ともMC勝者)**:

| | 第1 | 第2 | 第3 | 第4 | 第5 |
|---|---|---|---|---|---|
| v1.5.0(特典あり)MC制覇 | 18.9% | 43.7% | 55.7% | 59.3% | 62.7% |
| 特典オフ | 16.4% | 46.4% | 55.7% | 59.7% | 58.0% |
| v1.5.0 引退時カラ | 71 | 84 | 90 | 95 | **98** |
| 特典オフ 引退時カラ | 71 | 85 | 89 | 90 | **90** |

カラは確実に +8 届く一方、MC 制覇率の上昇は第5世代 +4.7pt に収まった(誤差 ±2.5pt)。**「両親とも」条件のおかげで王者血統率は第5世代でも 45% 程度**にとどまり、特別感が保たれている(片親条件だと 85% に達して特典が常態化してしまう)。

### レコード・通算勝利数(v1.2)

- スロット単位で `records[raceId] = {time, name, year, ground}`(自己ベスト)と `raceWins[raceId]`(通算勝利数)を蓄積
- `applyRaceResult()` で更新。既存レコードを塗り替えると結果画面+日誌に「🏅 コースレコード更新!」(初出走は黙って基準を記録)
- 表示は 📅「レースと記録」モーダルの「🏅 これまでの記録」タブ(v1.4 でタブ化)— **出走したレースだけ** 並び、タイム+地面タグ+保持マイマイ+樹立年を表示

### 画面まわりの決めごと(v1.4)

スマホで「大事なものが下にあってスクロールしないと届かない」問題をまとめて解消した回。触るときは同じ原則を守ること。

- **きゅう舎日誌は scr-main の最上部パネル**。調教・レースの結果コメントをスクロールなしで読むため。`renderLog()` は `G.logs` を**逆順**に描画し `scrollTop = 0`(配列自体は従来どおり末尾 push・先頭 shift のまま)
- **配合の親えらび画面は `#breed-bar` を `position: sticky; top: 0`** に固定。選択状況と「この2頭で配合!」が、殿堂カードを何枚スクロールしても常に押せる
- **📅「レースと記録」はタブ制**(`showCalendarTab('calendar' | 'records')`)。カレンダー12行と記録表を縦積みすると記録がずっと下になって見えなかったため。モーダルを開くときは必ずカレンダータブから始める
- **`closeModal('modal-confirm')` は `G._confirmYes` を null にする**(枠外タップやEscで閉じたとき保留処理が残らないように)。そのため `btn-confirm-yes` は **closeModal より先にコールバックを取り出す**必要がある — 順序を入れかえると「はい」が無反応になるので注意

## セーブ仕様(重要)

`localStorage["starMaimai.slot0" | "slot1" | "slot2"]` に JSON:

```
{ version: 4,            ← SAVE_VERSION
  money, year,
  hall: [殿堂エントリ...],           ← 各エントリに gen / parents / canBreed / royal
  current: マイマイ|null,            ← gen / parents / royal を含む
  records:  { raceId: {time, name, year, ground} },  ← レース自己ベスト (v3)
  raceWins: { raceId: 勝利回数 },                     ← レース通算勝利数 (v3)
  updatedAt }
```

**マイグレーション**: `MIGRATIONS = { 旧version: (data)=>新data }` のチェーン方式。`loadSlot()` が読み込み時に `migrateSave()` を通し、変換後は即再保存する。

- `MIGRATIONS[1]`(v1→v2): current / hall の全個体に `gen: 1` `parents: null` を付与
- `MIGRATIONS[2]`(v2→v3): `records: {}` を追加し、`raceWins` は殿堂+現役の `results` の1着から復元(タイムは過去に保存していないため復元不可、以後の出走から記録)
- `MIGRATIONS[3]`(v3→v4): 全個体に `royal: false` を付与。**既存個体は遡って判定できない**(`parents` は名前しか持たず、同名個体もありうるため)。以後に配合された子から正しく付く

> ⚠ **セーブ構造を変えるときの手順**: (1) `SAVE_VERSION` を +1、(2) `MIGRATIONS[旧番号]` に変換関数を追加。これを守れば公開後のアップデートでプレイヤーのデータが消えない。

## バランス設計と検証

調整パラメータは **`TUNE` オブジェクトに全集約**(初期値倍率・成長・疲労・エサ・レース物理・NPC強度以外に `GRADE_NPC` / `GRADES` / `RACES` / `MONTH_WET`)。

v1.0.0 決定値は、実コードをヘッドレスで 500シーズン×3方針 自動プレイして調整した:

| プレイスタイル | G3勝率 | G2勝率 | G1勝率 | MC制覇/シーズン |
|---|---|---|---|---|
| 雑(ランダム調教・カルシウムなし) | 63% | 29% | 5% | 1.4% |
| 丁寧(最低ステ調教+カルシウム) | 65% | 39% | 9% | 3.6% |
| 上級(ぬめり体力集中+カルシウム) | 73% | 55% | 22% | 19% |

設計意図 =「初代でも運が良ければ勝てる。うまく育てれば数世代で届く」(ユーザー指定)。

**世代交代バランス(v1.1.0 決定値、400試行×4世代)** — 親は殿堂のベスト2(足りなければ+野生):

| プレイスタイル | 第1世代 | 第2世代 | 第3世代 | 第4世代 |
|---|---|---|---|---|
| 上級(ぬめり体力集中+カルシウム) MC制覇/シーズン | 22% | 55% | 64% | 68% |
| 丁寧(最低ステ調教+カルシウム) MC制覇/シーズン | 1.5% | 8.5% | 13% | 17% |

**遺伝刷新の検証(v1.3.0 決定値、300試行×5世代)** — 上級方針、親は殿堂のベスト2:

| MC制覇/シーズン | 第1世代 | 第2世代 | 第3世代 | 第4世代 | 第5世代 |
|---|---|---|---|---|---|
| 旧 v1.2.1(単純平均×15%) | 18.5% | 45.0% | 60.5% | 58.5% | 65.5% |
| 新 v1.3.0(優勢親+プロファイル配分) | 18.0% | 48.3% | 56.7% | 57.3% | 60.3% |

誕生時ステ合計も 112/145/153/154/157 とほぼ一致 → **体感だけ強化してインフレなし**を確認。

**「遺伝率を上げるだけ」案は却下(2026-07-21)**: `statInheritRate` を 0.15→0.25 にすると第3世代で MC 制覇 80%、0.35 で 93% とはっきりインフレする。apt の平均回帰(`aptRegress`)を 0.5〜0.6 に強め `gfParentW` を下げて隠しステ側で相殺しても第5世代 88〜91% までしか戻らず、**打ち消せない**。理由は「ステは調教でどうせ上限120近くまで仕上がるので、初期値の上乗せがほぼそのまま勝率に効く」ため。**遺伝感は量ではなく配分と見せ方で出すのが正解**、という結論。

**ステ上限 120×4 は理論上ありうる(2026-07-21 検証)**: ボトルネックは調教ではなく **カラ**。カラは特選カルシウム(+4/ターン)でしか伸びず、その資金(500G/ターン)を稼ぐレースのターンとエサのターンが食い合うため。

- 必要条件は **紫サイコロ2個の合計が11以上**(カラ初期39以上)。10以下だと他が全部6でも apt 全部1.7でも**数学的に不可能**
- 到達プラン: レースを3回(新芽ステークス→さくらカップ→梅雨グランプリ)に絞り、残り21ターンすべて特選カルシウム
- **apt はほとんど効かない**(全部1.0でも到達可能)。調教は必要量に対して余力が大きい
- 発生率(最適プレイ+全レース1着を仮定した上限): 野生スタートで 0.5%(調教ロールが毎回最大の場合)、平均ロールならほぼ 0%。120×4 の親から配合すれば平均ロールで約6%、好ロールで約41% → **実質「配合を重ねた先のご褒美」**

**「レースに出ず調教だけした個体を種親にする」搾取の検証済み(2026-07-19)**: 引きこもり調教でステ合計310〜423の親は作れるが、遺伝率15%で子への上乗せは僅か+5pt程度。一方でそのシーズンの賞金・タイトルを丸ごと失うため、**2シーズン合計では出走し続ける方が明確に得** → 対策不要と判断(理由: 遺伝率15%の希釈、apt/gf は調教で盛れない、ステ上限120、賞金喪失)。

テストハーネスはセッションの scratchpad にあった `simtest.js` / `flowtest.js` / `balance.js`(v1.0)、`genesim.js` / `uitest.js` / `maxstat.js`(v1.3)。いずれも **DOM スタブ + `vm` で index.html の `<script>` をそのまま実行し、`doTrain`/`setupRace`/`applyRaceResult`/`makeSnailFromDice` など実関数で回す方式**。再調整時は同方式を再作成すると早い。ハーネス作成時のつまずきどころ:

- `showScreen()` が `window.scrollTo` を呼ぶので sandbox に `scrollTo(){}` が要る
- `persist()` を止めるには `G.slot = -1` にする
- レースでターン24に達したときは `retireSnail()` が呼ばれないので、ハーネス側で明示的に呼ぶ
- `$()` が毎回新しいスタブを返すと `innerHTML` を読み返せない → id ごとに要素をキャッシュするスタブにすると出力を検証できる

UI 確認はヘッドレス Chrome(`--headless=new --screenshot --virtual-time-budget`)。`localStorage.setItem` する `<script>` を本体スクリプトの手前に差し込んでセーブを注入し、末尾に `startBreedScreen()` → `toggleBreedSel()` → `confirmBreed()` → `finishDiceRoll()` を叩く駆動スクリプトを足して、`document.title` に `window.onerror` の件数を出して確認する方式。
